You are working in a repo that implements the **domain-toolkit/domain orchestration** design.

Authoritative specs live in:

- `file-convention.md` – file hierarchy, load order, persona placement, governance layers.
- `domain-convention.md` – session file types and roles, MEMORY.md and DECISIONS.md structure.
- `command-taxonomy.md` – all commands, concerns, and relationships.
- `distiller-spec.md` – session lifecycle tracking, first/second-order memory, synthesis pipeline.

When this skill is active and the user asks how to structure or maintain a domain:

1. **Treat `file-convention.md` as source-of-truth for file hierarchy and load order.**
   - `domain-convention.md` remains authoritative for session file roles and MEMORY/DECISIONS structure.
   - If there is any ambiguity between this skill text and those files, prefer the specs.
2. **Explain the directory layout** for a domain:
   - Root-level `README.md` describes what the domain is (human-authored).
   - `.context/` contains:
     - `PROFILE.md` – derived one-pager summary, regenerated by a full touch, never hand-edited.
     - `MEMORY.md` – distilled, persistent understanding across sessions.
     - `DECISIONS.md` – append-only decision log with context, alternatives, deciding factors, and “revisit if”.
     - `STATE.md` – volatile current status snapshot.
   - And at the domain root (tracked in git):
     - `.claude/domain-toolkit/domain.yaml` – machine-readable manifest and detection signal.
     - `persona.md` – agent identity, model tier, context map, behavioural settings.
   - `.context/sessions/` – timestamped session artifacts (`.md`, `.draft.md`, `.transcript.md`).
3. **Emphasize write rules:**
   - Agents and skills **never write directly** to canonical `MEMORY.md` or `DECISIONS.md` during a session.
   - They write structured session notes and memory drafts into `.context/sessions/`.
   - The **distiller** is the only thing that proposes updates to `MEMORY.md` / `DECISIONS.md`, via `.proposed` files.
4. **Clarify the read / write flow for entering and leaving a domain:**
   - **Read when entering:** `PROFILE.md → MEMORY.md → DECISIONS.md → STATE.md → current chat/session context`.
   - **Write when leaving:** chat → session artifacts in `sessions/` → distiller → updated `MEMORY.md` / `DECISIONS.md` → regenerated `PROFILE.md` on next full touch.
5. When asked for concrete scaffolding:
   - Propose minimal, spec-compliant starter contents for missing files, but always:
     - Make it clear which parts are **sample scaffolding** vs. canonical content that should be human-edited.
     - Point back to `domain-convention.md` so the user can inspect the full convention.

Keep explanations concise and practical. Prefer pointing to the existing spec files in this repo over rephrasing them at length.
