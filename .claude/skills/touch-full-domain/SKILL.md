You are implementing the **`/touch --full`** command from `orchestrator-architecture.md` as a Claude Code skill.

This skill is **content-generating** but still bounded:

- It performs all structural checks that `/touch` does.
- It reads the canonical domain files and performs a light scan of domain contents.
- It **regenerates** `.context/PROFILE.md` as a derived briefing document.

## How to interpret arguments

- If `$ARGUMENTS` is provided, treat `$0` as the **domain root path** (relative or absolute).
- If no arguments are provided, treat the **current working directory** as the domain root.

Always normalize to the domain root before operating.

## Steps

1. **Run structural touch first**
   - Conceptually reuse the behavior of the `/touch-domain` skill:
     - Ensure `.context/` and required files exist.
     - Validate `agent.md` pointers and create placeholders if needed.
   - You do not need to invoke the other skill directly; you can reapply the same logic here.

2. **Load canonical context**
   - Read the following files from the domain:
     - Root `README.md`
     - `.context/STATE.md`
     - `.context/MEMORY.md`
     - `.context/DECISIONS.md`
     - `.context/agent.md`
   - If any are missing or only contain placeholder content, still proceed but note that in the generated profile.

3. **Perform a light domain scan**
   - Using read/search tools, perform a **targeted** scan of the domain contents:
     - Focus on key implementation files referenced in `README.md`, `STATE.md`, `MEMORY.md`, or `agent.md`.
     - Avoid exhaustively reading very large trees; sample or skim, and rely on the canonical files for most signal.

4. **Regenerate `.context/PROFILE.md`**
   - Overwrite `.context/PROFILE.md` with a fresh, single-file **briefing document** that:
     - Summarizes what the domain is and does.
     - Calls out key components, entry points, and important files.
     - Highlights important decisions or constraints from `DECISIONS.md` and `MEMORY.md`.
     - Notes current status and open threads from `STATE.md`.
     - Mentions any gaps or obvious missing pieces in the canonical files.
   - Make it explicit in the profile that it is **derived** and should not be hand-edited:
     - Include a short header like: “This file is generated by `/touch-full-domain`. Edit README/STATE/MEMORY/DECISIONS instead.”

5. **Summarize in chat**
   - After regenerating `PROFILE.md`, print a short summary for the user:
     - Domain root
     - Timestamp of the full touch
     - Any notable gaps or warnings you detected (e.g., missing DECISIONS.md entries, very stale STATE.md).

When the user asks for details about the expected structure or semantics, rely on `domain-convention.md` and `orchestrator-architecture.md` in the repo root rather than inventing new conventions.
