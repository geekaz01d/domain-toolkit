okay so lets walk through what happens here. a session is staged. what if i continue that session?

## Assistant

Good catch. If you continue a session (`claude --continue`), the hook fires again with the same `session_id` but `source: "resume"`. Two things happen:

1. **The hook** appends a second entry to `session-index.jsonl` with the same session ID — harmless, the stager dedupes by session ID
2. **The JSONL keeps growing** — Claude appends to the same `<session_id>.jsonl` file. But the transcript was already staged from the first run, so the new content is never picked up

That's the problem. The staged `.transcript.md` is a snapshot from when the stager ran, not the final state. A continued session's later turns are silently lost.

Options:

**A — Re-stage on change**: The stager compares the JSONL's mtime against the staged transcript's mtime. If the JSONL is newer, re-extract and overwrite. Simple, but means the transcript file isn't stable — downstream consumers (distiller) could read a half-written file.

**B — Append-only staging**: Each resume creates a new transcript file (`*-part2.transcript.md`). The distiller sees all parts. But this fragments what's logically one session.

**C — Stage only finished sessions**: Don't stage until the session is truly done. But "done" is hard to detect — there's no reliable `SessionEnd` signal for crashes.

**D — Always re-extract, atomic write**: Stager always re-extracts from the JSONL (it's the source of truth), writes to a temp file, then atomically renames. The `.staged-sessions` file tracks session ID + JSONL size at time of staging. If the size grew, re-stage.

I'd lean **D** — the JSONL is always the source of truth, the staged transcript is a derived artifact that gets refreshed when the source changes. Cheap to implement too — just track `session_id:file_size` instead of bare `session_id`.

What do you think?
