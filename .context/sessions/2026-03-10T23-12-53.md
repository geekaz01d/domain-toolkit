## Checkpoint: 2026-03-10T23:12:53Z

- **Scope**: SessionStart hook implementation, workspace auto-launch, and auto-run-command extension integration.

- **Since last checkpoint**:
  - Created `hooks/session-start.sh` in the firehose repo — template SessionStart hook that reads `.context/agent.md` + PROFILE/MEMORY/DECISIONS/STATE on session start.
  - Installed hook globally via `~/.claude/settings.json` (delegates to repo copy).
  - Verified hook fires on session resume — all context files injected successfully.
  - Removed the "Start Domain Session" task from `firehose.code-workspace` (was launching terminal `claude` on folderOpen — unwanted).
  - Investigated auto-launching Claude Code extension on workspace open. VS Code/Cursor has no native mechanism to fire arbitrary extension commands on startup.
  - Evaluated `gabrielgrinberg/auto-run-command` extension (MIT, ~90 lines, built 2018, unmaintained). Decision: use now, fork later.
  - Built VSIX from source (required TS fix and Travis badge removal), installed in Cursor.
  - Added `auto-run-command.rules` to workspace settings to fire `claude-vscode.newConversation` on open.
  - Added `extensions.recommendations` to workspace file for `anthropic.claude-code` and `gabrielgrinberg.auto-run-command`.

- **Decisions made**:
  - **SessionStart hook as global install**: Hook lives in `~/.claude/settings.json` with empty matcher (fires on all events: startup, resume, clear, compact). The script is a no-op if `.context/agent.md` doesn't exist, so non-domain repos are unaffected.
  - **auto-run-command for extension auto-launch**: Accepted dependency on a 2018 MIT extension to bridge the gap between "workspace opens" and "Claude session starts." Scoped via workspace settings — only domain kit workspaces include the auto-run rule.
  - **Fork deferred**: Use the marketplace/VSIX version now; fork and maintain under firehose namespace later. The extension is trivial enough that maintenance risk is low.
  - **No terminal claude on workspace open**: Removed the folderOpen task that launched `claude` CLI. The extension + SessionStart hook replaces it entirely.

- **Open questions**:
  - Does `claude-vscode.newConversation` conflict with Cursor's session restore? If Cursor restores a previous Claude tab AND auto-run fires newConversation, user may get two tabs. Needs testing.
  - Should the hook also fire on `compact` events? Currently it does (empty matcher = all events). This re-injects context after compaction, which is probably desirable but untested.
  - When forking auto-run-command, should we strip conditions entirely and just fire commands unconditionally? Or keep `hasFile` for checking `.context/agent.md` presence?

- **Current thinking**: The domain kit startup chain is now: workspace open -> context files as tabs -> auto-run fires newConversation -> SessionStart hook injects agent.md + context map. Three independent mechanisms (folderOpen task, auto-run extension, Claude Code hook) that compose cleanly. The auto-run-command dependency is the weakest link — small, MIT, but unmaintained. Fork is the right move once we have bandwidth.

- **Notes**: The VSIX build required: `npm install --ignore-scripts @types/vscode@1.50.0 typescript@latest`, fixing `resolve()` -> `resolve(undefined)` in run-shell-command.ts, removing Travis CI SVG badge from README, then `npx tsc -p ./ --skipLibCheck && npx @vscode/vsce package --no-dependencies --no-yarn --skip-license`.
