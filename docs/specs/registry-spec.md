# Registry Spec

**Status:** Draft — captured from working session 2026-03-18
**Context:** Defines the derived domain registry — how it's built, what it contains, and how it's consumed. The registry is the primary input for overview/sweep, `open-domain` (set resolution), and `touch-domain --all`.

---

## Design Principles

1. **The registry is derived, not authored.** It is built by scanning `domain.yaml` files across the filesystem. Never hand-edited.
2. **The filesystem is the source of truth.** `domain.yaml` files in each domain are the canonical source. The registry is a computed view.
3. **Merge, not replace.** domain.yaml has user-declared fields that are preserved across updates. The registry reflects whatever domain.yaml says.
4. **Discovery is explicit.** Scan paths are onboarded via CLI. The registry only knows about paths it's been told to look at.

---

## Location

`~/.claude/domain-toolkit/REGISTRY.yaml` — local, machine-specific, synced via Syncthing across viewport nodes.

Replaces the previous `REGISTRY.md` markdown table format.

## Discovery

### Scan Paths

The registry scanner needs to know where to look for domains. Scan paths are directories that may contain domains (directly or nested).

Scan paths are managed via CLI:

```bash
add-domain --scan-path ~/sources
add-domain --scan-path ~/sources/fluffy_sources
add-domain --scan-path ~/sources/infrastructure
```

Scan paths are stored in the registry file itself (`REGISTRY.yaml`), under the top-level `scan_paths` key. `add-domain --scan-path` appends to this list; `add-domain --update` reads it to know where to walk.

```yaml
# In REGISTRY.yaml:
scan_paths:
  - ~/sources
  - ~/sources/fluffy_sources
  - ~/sources/infrastructure
```

### How Scanning Works

`add-domain --update` walks each scan path, looking for directories that contain `.claude/domain-toolkit/domain.yaml`. Each found domain.yaml is read and its contents are merged into the registry.

Scanning is not recursive into domains — once a domain.yaml is found, its subtree is not scanned for nested domains. Grouped domains (like `infrastructure/systems-*`) are found because the parent directory is a scan path.

### Explicit Addition

`add-domain <path>` adds a single domain to the registry by reading its domain.yaml. This works regardless of whether the path is under a scan path. Useful for one-offs or domains in unusual locations.

`add-domain --new <path>` scaffolds a new domain (creates domain.yaml, .context/, CLAUDE.md, etc.) and adds it to the registry in one step.

## Registry Format

YAML. Machine-generated, human-readable, diffable, easy to query programmatically.

```yaml

# Do not edit — regenerate from domain.yaml files on disk

generated: 2026-03-18T14:30:00
scan_paths:
  - ~/sources
  - ~/sources/fluffy_sources

# ── Domain Entries ────────────────────────────────────────────────
```

## Domain Entry Fields

Each domain entry in the registry mirrors the fields from its `domain.yaml`. The registry does not add or compute additional fields beyond what domain.yaml provides, with one exception:

| Field | Source | Description |
|-------|--------|-------------|
| `path` | Discovered at scan time | Absolute path where domain.yaml was found. Machine-specific. |
| All other fields | Copied from domain.yaml | name, repo, type, description, sets, canonical_source, default_branch, remotes, kit_health, last_touched, plus any freeform keys. |

The `path` field is the one field the registry adds — domain.yaml doesn't declare its own path (it's implicit), but the registry needs to record where it found each domain.

---

## Sets Index

A computed reverse lookup appended to the registry. Maps each set name to the list of domains that belong to it.

Built automatically from the `sets` field of each domain entry. Used by `open-domain` for fast set lookup without scanning every domain entry.

---

## Consumers

| Consumer | What it reads | Purpose |
|----------|--------------|---------|
| **`open-domain <name>`** | Single domain entry by name | Resolve domain name to path for viewport launch |
| **`open-domain <set-name>`** | Domain lookup, then sets index | Open all domains in a named set (if no domain matches the name) |
| **`touch-domain --all`** | All domain entries | Iterate for batch touch |
| **`overview / sweep`** | All domain entries + personal domain profile | Capacity-aware briefing. Reads `last_touched` for currency signal. |
| **`group-domain <set-name>`** | Sets index + domain entries | List members, add/remove domains from sets |
| **`add-domain --update`** | Entire registry (to merge) | Refresh derived fields, preserve structure |

---

## Name Resolution

Domains and sets share a namespace. Any command that accepts a name (e.g. `open-domain`, `touch-domain`) resolves it using this order:

1. **Domain name** — check `domains:` entries. If a domain with this name exists, use it.
2. **Set name** — check `sets:` index. If a set with this name exists, use it.

If both a domain and a set share a name, the domain wins. `touch-domain` should surface name collisions as a concern.

`group-domain` operates exclusively on sets:

```
group-domain infrastructure --add systems-harrklen
group-domain infrastructure --remove systems-harrklen
group-domain infrastructure       # list members
group-domain --list               # list all sets
```

Under the hood, `group-domain` modifies the `sets` field in each affected domain's `domain.yaml`, then triggers a registry rebuild. The source of truth for set membership is always domain.yaml.

---

## Currency Signal

`last_touched` in the registry reflects the date of the last `touch-domain` run on each domain. This gives overview/sweep a lightweight signal for domain currency without needing to open each domain and inspect its contents.

A domain that hasn't been touched in weeks is a candidate for attention. A domain touched today is likely current. Sweep can sort and prioritise by this field.

## Lifecycle

### Initial Setup

```bash
# Tell the registry where to look
add-domain --scan-path ~/sources
add-domain --scan-path ~/sources/fluffy_sources

# Build the registry
add-domain --update
```

### Adding a New Domain

```bash
# Scaffold and register in one step
add-domain --new ~/sources/my-new-project

# Or register an existing domain
add-domain ~/sources/existing-project
```

### Refreshing the Registry

```bash
# Re-scan all scan paths, update derived fields
add-domain --update
```

### Missing Domains

If a domain was previously in the registry but its path is unreachable on the next `add-domain --update` (unmounted drive, domain on another machine), it is **preserved** with `on_disk: false` added to its entry. The registry remembers domains even when their disk isn't mounted — the user may have multiple machines syncing a shared registry via Syncthing.

Domains with `on_disk: false` are still valid registry entries. They resolve by name, appear in sets, and show up in `overview`. Commands that need filesystem access (e.g., `touch-domain`, `open-domain`) will report that the path is unreachable.

When the domain reappears on disk (drive mounted, machine reconnected), the next `add-domain --update` removes the `on_disk: false` flag and refreshes the entry from domain.yaml.

### Excluding a Domain

To explicitly exclude a domain from the registry without deleting its domain.yaml, add a field to domain.yaml:

```yaml
registry: exclude
```

`add-domain --update` skips domains with `registry: exclude`.
